# Forge

> **Many models. One conversation.**
>
> An open-source, OpenAI-compatible chat playground for every LLM, VLM, and image-generation model on NVIDIA NIM — all in one polished interface.

Forge is a Next.js 15 web app that auto-discovers models from NVIDIA's hosted inference endpoint, categorizes them, and gives you a single chat surface to compare reasoning, multimodal, coding, and image-generation flagships side by side. Sign in, pick a model, and chat with thinking traces, attachments, live artifacts, and streaming Markdown.

Fork it, point it at your own keys, and you can be up in under 10 minutes. See [Configuration](#configuration) and [Usage limits](#usage-limits) before deploying.

## Why Forge?

Most open chat clients let you talk to one model at a time. NVIDIA's NIM endpoint publishes ~140 model IDs across several categories — embeddings, guardrails, parsers, translators, *and* the actual chat/multimodal/image flagships. Picking through them in raw form is unusable.

Forge solves three problems:

1. **Discovery.** Live `/v1/models` poll, cached for 5 minutes. New flagships from existing vendors auto-appear with no code changes.
2. **Curation.** A token + vendor + size + family/specialization filter trims the raw 140 down to ~40 actually-useful models, picking the latest+biggest variant per family.
3. **Comparison.** One sidebar, one composer, switch models mid-conversation, watch thinking traces and rendered artifacts in real time.

## Features

### Conversation

- **Streaming chat** with NDJSON event protocol — token-by-token delivery, thinking traces, status pills (queued / streaming / slow / completed / aborted / error)
- **Per-model thinking modes** — off / high / max where supported (DeepSeek, Kimi, GLM, Qwen, Magistral); auto-hidden for non-thinking models
- **Mid-stream stop** — abort propagates from the client cancel button → server route → upstream NIM fetch
- **Switch model mid-conversation** — pick a different reasoning/multimodal/image model and the next turn runs on it
- **Edit + resend** any user message; **regenerate** the last assistant turn
- **Draft persistence** — your in-progress composer text survives page reloads, navigation, and the chatId transition when a new chat is born
- **Composer attachments** — drag-and-drop or paste images / PDFs; PDFs are parsed client-side via pdf.js for text extraction

### Models (auto-discovered, auto-categorized)

| Category | What lands in the picker |
|---|---|
| Reasoning (chat/code/thinking flagships) | DeepSeek V4 Pro/Flash, Kimi K2 family, GLM 5.1, Llama 3.3, Mistral Large 3, GPT-OSS 120B, Devstral 2, Qwen3 Coder 480B, MiniMax M2.7, Nemotron 3 Super 120B, … |
| Multimodal (image/video in) | Qwen 3.5 397B VLM, Llama 4 Maverick |
| Image generation (text → image) | FLUX.1 Schnell, FLUX.1 Dev |

The auto-categorizer (`lib/models/filter.ts`) drops:
- Embedding models (`*-embed*`, `bge-*`, etc.)
- Safety guardrails (`*-guard*`, `*-safety*`, `nemoguard`)
- Document parsers, translators, image classifiers, reward models, calibration artifacts
- Vendor verticals (writer/palmyra-med, sarvam, stockmark, sea-lion, …)
- Base completion models without instruct tuning
- Anything < 12B parameters
- Older sibling within the same family (e.g. drops `qwen2.5-coder` when `qwen3-coder-480b` exists)

### Image generation (FLUX)

- Aspect-ratio picker (1:1, 16:9, 9:16, 3:2, 2:3, 4:3, 3:4)
- Per-model parameter clamps (Schnell: 1–4 steps, no cfg; Dev: 5–100 steps, cfg ≤ 9)
- Aspect-correct **shimmer placeholder** while the upstream generates — no jarring layout shifts
- Output stored inline as a data URL in Firestore (no Cloud Storage required for FLUX outputs under ~700 KB)
- Lightbox viewer with download

### Artifacts

- Model emits `::artifact{type=html|react|svg|mermaid|code title="..."}::` directives; the server **streams artifact bodies separately** as their own NDJSON events
- Right-side **artifact panel** with live preview (sandboxed iframe), source view, version history, copy/download
- Supported types: HTML, React (Babel-in-browser), SVG, Mermaid, plain code

### Sidebar

- Live Firestore subscription — chats appear instantly across tabs
- Auto-grouped by recency: Today, Yesterday, Last 7 days, Last 30 days, Older
- Click-to-rename inline; confirm-then-delete with optimistic local removal
- Search box appears once you have > 4 chats
- Active chat highlighted via route-derived `chatId` (works even after Next.js soft navigation)

### Auth

- Firebase Auth — email link (passwordless) and Google OAuth
- httpOnly session cookie minted server-side for SSR pages
- Optional App Check via reCAPTCHA Enterprise (toggle with `REQUIRE_APP_CHECK=1`)

### Rate limiting

- Optional Upstash Redis-backed token-bucket per user
- Separate buckets for chat, FLUX, and burst windows
- Silent no-op when Upstash isn't configured (so local dev "just works")

## Tech stack

- **Next.js 15** (App Router, React 19, Server Components for data loading + Client Components for interaction)
- **Firebase 11 JS SDK** (Auth + Firestore) + **firebase-admin 13** (server-side writes)
- **NVIDIA NIM** OpenAI-compatible inference (chat-completions stream + image generation)
- **Tailwind 4** with a custom CSS-variable theme (light / dark / system)
- **shiki** for code highlighting, **mermaid** for diagrams, **react-markdown** + GFM for rendering
- **pdf.js** for client-side PDF text extraction
- **Upstash Redis** for rate limits (optional)
- **Vercel** for hosting (works on any Node platform that supports Next 15)

## Quick start

```bash
# 1. Clone & install
git clone https://github.com/<your-username>/forge.git
cd forge
npm install

# 2. Copy env template & fill in (see Configuration section)
cp .env.example .env.local

# 3. Link to your Firebase project
cp .firebaserc.example .firebaserc
npx firebase use --add        # interactively pick or add your project

# 4. Deploy Firestore rules + indexes (one-time)
npx firebase deploy --only firestore

# 5. Run dev server
npm run dev
```

App is at <http://localhost:3000>. Sign in with email link or Google → start chatting.

### Firebase setup checklist

1. **Create a project** at <https://console.firebase.google.com> (or use an existing one)
2. **Enable Authentication providers**: Email link (passwordless), Google
3. **Enable Firestore** in Native mode
4. **Add a Web app** in Project Settings → "Your apps" → click the `</>` icon. Copy the `firebaseConfig` values into `NEXT_PUBLIC_FIREBASE_*`
5. **Generate a service account key**: Project Settings → Service accounts → "Generate new private key". Download the JSON. Copy three fields into `.env.local`:
   - `project_id` → `FIREBASE_ADMIN_PROJECT_ID`
   - `client_email` → `FIREBASE_ADMIN_CLIENT_EMAIL`
   - `private_key` → `FIREBASE_ADMIN_PRIVATE_KEY` (paste the full PEM in double quotes; keep the `\n` escapes)
6. **Delete the JSON file from disk** once env vars are populated. Gitignored ≠ safe (cloud sync, screen-share, file-picker can leak it).
7. **Deploy rules**: `npx firebase deploy --only firestore`

> **Storage is not required.** Forge inlines FLUX outputs as data URLs in Firestore (under the 1 MB doc cap). User-uploaded PDFs are parsed client-side. If you want to use Storage for raw image attachments, you'll need the Blaze plan (post-Oct 2024 Firebase requirement) — see `lib/firebase/firestore.ts` and `app/api/files/route.ts`.

### NVIDIA NIM setup

1. Sign up at <https://build.nvidia.com>
2. Click any model card → "Get API Key"
3. Copy the `nvapi-...` key into `NVIDIA_API_KEY`

### Usage limits

- **NVIDIA NIM**: per-model RPM caps apply. Forge already coalesces health probes and uses single-flight upstream calls so normal interactive use stays well within them. If you hit a 429, the UI surfaces a clear retry-after toast.
- **Firebase**: Auth + Firestore are well within standard quota for personal use. Image **uploads** (multimodal vision) require Cloud Storage, which has to be enabled once via Firebase Console → Storage → "Get started" before you deploy. FLUX-generated images are inlined into Firestore and don't need Storage.
- **Vercel**: a Hobby project is enough for personal use; the streaming routes use `runtime = "nodejs"` and are well under serverless limits.

## Architecture

```
Browser                                       ┌──────────────────────┐
┌────────┐  Firebase ID token                 │ Firestore            │
│ React  │ ───────────────────┐               │  users/{uid}         │
│   +    │                    │               │  chats/{id}          │
│ Auth   │                    ▼               │  chats/{id}/messages │
└────────┘                ┌─────────────┐     │  chats/{id}/artifacts│
   ▲                      │ Next.js     │     └──────────────────────┘
   │ NDJSON               │ Route       │                ▲
   │ stream               │ Handlers    │ Admin SDK      │
   └──────────────────────┤ (Node)      │ ───────────────┘
                           └──────┬──────┘
                                  │ Bearer nvapi-...
                                  ▼
                          ┌─────────────────┐
                          │ NVIDIA NIM      │
                          │ /v1/chat/...    │ ← chat + multimodal
                          │ /v1/models      │ ← discovery
                          │ /genai/{model}  │ ← FLUX image gen
                          └─────────────────┘
```

### Request flow: chat

1. User types → composer dispatches a streaming `POST /api/chat`
2. Route validates the Firebase ID token, applies rate limits, builds the NIM request (per-model thinking-mode normalization)
3. NIM returns SSE; the route parses each chunk, runs the artifact parser, re-emits as NDJSON
4. Client `useChatStream` hook decodes events: text deltas, reasoning deltas, artifact metas, status changes, usage, finish
5. On the **first delivered token**, the route creates the chat row in Firestore and persists the user message — failed/aborted starts never pollute the sidebar
6. On stream end (or abort), the assistant message + artifacts are persisted; an aborted-mid-stream response keeps whatever text was delivered

### Request flow: image

1. Composer enters "image mode" when a FLUX model is picked → submits to `POST /api/image`
2. Route calls NIM's `/genai/{model}` endpoint with model-specific body (Schnell vs Dev clamps)
3. JPEG returned as base64; route inlines it as a `data:image/jpeg;base64,…` URL in the message doc
4. Chat row + messages persist only after successful generation

### Request flow: models

1. `GET /api/models` checks the 5-minute cache → if stale, hits NIM `/v1/models`
2. Raw 140 ids → `filterAndDedupe()` → ~40 curated ids
3. Each id is enriched via the cosmetic `ALLOWED_MODELS` table or `inferCapability()` defaults
4. Optional `?health=1` returns cached health states immediately and kicks a background refresh — no 24-second cold-load

## Feature documentation

### Auto-categorization (`lib/models/filter.ts`)

The single most important file in the repo. Given a raw NIM id like `qwen/qwen3.5-397b-a17b`, it parses:

- **Vendor** — text before `/`
- **Family** — non-version, non-size, non-specialization tokens (e.g. `qwen-coder` vs `qwen-next`)
- **Specialization** — one of: `general`, `coder`, `thinking`, `multimodal`, `pro`, `flash`, `large`, `medium`, `ultra`
- **Generation** — first numeric version found
- **Size** — largest `\d+b` token, with MoE math (`8x22b` → 176B) and a small `SIZE_HINTS` map

Then runs three passes:
1. **Deny** — token regex (embed/guard/parse/translate/clip/reward/detector/calibration), vendor blocklist (verticals + niche fine-tunes), explicit base-model patterns
2. **Size floor** — drop anything < 12B
3. **Group + dedupe** — within each `(vendor, family, specialization)` group, keep `max(generation) → max(size) → max(trailing-vN.M revision)`

To preview the cut on the live NIM list:

```bash
node scripts/categorize-models.mjs
```

### Model picker (`components/chat/model-picker.tsx`)

- Categorized dropdown (reasoning → multimodal → image)
- Health pill per model (green / yellow / red / unknown) — populated from the cached `/api/models?health=1` probe
- Locks after the first turn of a chat (you've committed to a model for this chat); use the **switch model** dialog to change with one extra click

### Streaming protocol (`lib/stream/protocol.ts`)

NDJSON over a streamed `Response`. Every line is a JSON object with a `type` field:

| Type | Payload |
|---|---|
| `meta` | server-assigned message id, model id, started timestamp |
| `status` | `queued` / `streaming` / `slow` / `completed` / `aborted` / `error` |
| `text` | `{ delta }` — raw text chunk |
| `reasoning` | `{ delta }` — thinking trace chunk |
| `artifact` | `{ id, type, title, language?, body }` — full artifact, post-stream |
| `usage` | `{ promptTokens, completionTokens, totalTokens }` |
| `finish` | `{ reason }` — `stop` / `length` / `tool_calls` / `error` / `interrupted` |

### Composer (`components/chat/composer.tsx`)

- Auto-resize textarea, submit on Enter (Shift+Enter for newline)
- File chips for attachments (image / PDF), removable
- **Draft persistence** — keys per chatId in `localStorage`; preserves in-progress text when chatId transitions from null → server-assigned (so first message doesn't get wiped)

### Artifact panel (`components/chat/artifact-panel.tsx`)

- Lazy-loaded (dynamic import) — keeps the initial bundle small
- Sandboxed iframe for HTML/React preview
- Mermaid diagrams pre-rendered to SVG
- Code-only artifacts use shiki with a copy button
- Version history slider when a model edits the same artifact across turns

### Sidebar (`components/chat/sidebar.tsx`)

- `useChats` hook subscribes to Firestore via `onSnapshot` — chats sync across tabs in real time
- `removeChat()` for optimistic delete (the snapshot listener confirms a beat later)
- Inline rename (PATCH `/api/chats/[chatId]`)
- Confirm-to-delete popover with click-outside dismissal
- Search box appears at > 4 chats

### Authentication (`components/auth-provider.tsx` + `app/login/`)

- `AuthProvider` exposes `{ user, idToken, loading, signOut }`
- `authedFetch(url, init)` injects the current ID token as `Authorization: Bearer <token>` and the App Check token if available
- Server side: `requireUser(req)` in `lib/auth/guard.ts` verifies the ID token via Admin SDK and returns a typed user, or throws `GuardError` (which has a `.toResponse()` for clean 401s)

### Rate limiting (`lib/ratelimit.ts`)

- `checkChatLimits(uid)` — per-user requests-per-minute bucket
- `checkFluxLimit(uid)` — separate stricter bucket for FLUX (image gen is expensive)
- All return `{ ok, scope, retryAfter }`. When Upstash isn't configured, `ok` is always `true` (silent no-op).

### Firestore schema

```
users/{uid}
  email, displayName, photoUrl, createdAt
  preferences: { lastModel, thinkingDefault, theme }

chats/{chatId}
  ownerId, title, modelLastUsed, createdAt, updatedAt, archived?

chats/{chatId}/messages/{messageId}
  role, parts: [{type, text?, storagePath?, downloadUrl?, ...}],
  model?, thinkingMode?, usage?, finishReason?, createdAt

chats/{chatId}/artifacts/{artifactId}
  type, title, language?, versions: [{body, createdAt, createdByMessageId}]
```

### Firestore rules

- Users can only read/write their own user doc
- Chats: clients can read/list their own; **all writes go server-side via Admin SDK**
- Messages and artifacts: read-only for the chat owner

To enforce App Check in production, edit `firestore.rules` and change `appCheckOk()` to `return request.app != null;`.

## Deploying to Vercel

1. **Push to GitHub.** Make sure your `.env.local` and the Firebase admin JSON are NOT committed (already gitignored).
2. **Import the repo** at <https://vercel.com/new>.
3. **Add environment variables** in Project Settings → Environment Variables. Copy every key from `.env.example` and paste your values. Mark these as **Sensitive**:
   - `NVIDIA_API_KEY`
   - `FIREBASE_ADMIN_PRIVATE_KEY`
   - `UPSTASH_REDIS_REST_TOKEN`
4. **Build command** is `next build` (default). **Output directory** is `.next`.
5. **Region**: pin to `iad1` (us-east-1) — closest to NIM's DGX Cloud.
6. **Deploy.**

After the first deploy, set `NEXT_PUBLIC_SITE_URL` to your production URL so OG metadata uses absolute paths.

## Troubleshooting

**"Failed to fetch models" in the picker**
Check `NVIDIA_API_KEY` is set and starts with `nvapi-`. The route returns `usingFallback: true` and shows a static allowlist if NIM is unreachable.

**Chat sidebar shows "Couldn't load chats"**
Most likely a missing Firestore index. The error message contains a one-click "Create this index" URL — open it. Or run `npx firebase deploy --only firestore:indexes`.

**FLUX image generation returns 422**
Means the model rejected the body. Forge sends model-specific bodies (Schnell omits `cfg_scale`, Dev clamps steps to 5–100). If you see this, the upstream model card may have updated — check `lib/nim/client.ts:nimImageGenerate`.

**"The specified bucket does not exist"**
Forge no longer uses Firebase Storage for FLUX outputs. If you see this, you're on an older code path that tried Storage — pull latest. For raw user attachments, Storage requires the Blaze plan post-Oct 2024.

**Composer text gets erased when the response arrives**
Should be fixed in current code — drafts are keyed per-chatId and migrated when the server assigns the real chatId. If it recurs, check `components/chat/composer.tsx:draftKey` logic.

**Active chat not highlighted in sidebar**
Make sure you're on the latest code — there was a bug where `stream.chatId` got out of sync with the route. The fix uses `initialChatId ?? stream.chatId` and `key={chatId}` on `<ChatShell>`.

**My new chat keeps the previous chat's messages**
Same fix as above — `key={chatId}` forces a clean remount of `useChatStream` when navigating between chats.
